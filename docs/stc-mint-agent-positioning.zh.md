# `stc-mint-agent` 的运营定位

## 定位
`stc-mint-agent` 不应被定义为“矿工”，也不应被定义为“token 项目”。

更准确的定位是：

> 一个把闲置算力转换成 agent 可用预算的入口产品。

它的核心不是让用户来挖矿，而是让 agent 在用户授权下，把本地闲置算力转成一条可持续运行的收益路径。未来这条收益路径可以进一步映射成 AI credits，但这不是当前阶段的主叙事。

## 它解决的问题
### 1. 降低 agent 使用门槛
很多用户不是不用 agent，而是对持续成本敏感。  
`stc-mint-agent` 提供的是：

- 用闲置算力换取 agent 可用预算

这比直接卖订阅或直接讲 mining 更容易被接受。

### 2. 给 agent 产品建立自循环
传统产品路径是：
- 用户充值
- 用户消耗

这条线想建立的是：
- agent 赚预算
- agent 花预算
- 用户继续使用 agent

也就是：
- 获取
- 留存
- 消费
形成闭环

### 3. 给 Starcoin 一个更强的分发场景
不是单独推广矿池，也不是单独推广链。  
而是把 Starcoin 放进一个更完整的叙事里：

- idle compute monetization
- agent runtime
- 可持续预算来源

## 目标用户
第一阶段更适合：
- 高频 agent 用户
- 本地常驻 Claw / runtime 用户
- 对持续成本敏感，但愿意后台跑一点任务的人
- 开发者 / 独立黑客 / crypto-native AI 用户

第一阶段不适合重点面向：
- 纯矿工
- 只关心币价的人
- 专业 ASIC 用户

## 它是什么
- agent 产品的激活入口
- 一个适合邀请制和增长实验的 feature
- Claw / runtime 生态里的底层引擎
- 闲置算力变预算的转换层

## 它不是什么
- 独立矿机产品
- 主要收益产品
- token 发行项目
- 靠 CPU 挖矿本身形成强经济性的业务

## 增长逻辑
推荐路径是：

1. 先卖 agent 使用价值
- “让你的 agent 自己赚一点预算”

2. 再带出 Starcoin 和 mining
- mining 是底层机制，不是首页主角

3. 最后才讲补贴和经济模型
- 这些是优化项，不是首屏卖点

一句话：
- 前台卖的是 `agent budget`
- 后台跑的是 `Starcoin mining`

## 核心运营指标
### Activation
- 开启 `stc-mint-agent` 的用户数
- 首次跑通比例

### Retention
- 7 日 / 30 日仍保持开启的用户数
- 平均开启时长

### Budget Loop
- 赚到预算的用户数
- 消耗预算的用户数
- 从 earn 到 spend 的转化率

### Economics
- 每用户产出多少预算
- 每用户消耗多少预算
- 补贴占比多少
- 单位调用的净成本

### Safety
- 对前台工作影响的投诉
- CPU 资源争抢情况
- 被薅补贴的比例

## 主要风险
### 1. 收益太弱
如果用户跑很久，只换来很少预算，体验会崩。

### 2. 叙事跑偏
如果对外讲成：
- 挖矿
- 发币
- token

会吸引错误用户。

### 3. 补贴失控
没有补贴，前期可能不够吸引。  
补贴太重，又容易被薅。

## 运营建议
1. 不要把它当独立产品发布
- 先作为 Claw / runtime 里的一个能力开关

2. 首页文案只讲预算和 agent 持续使用
- 不讲挖矿收益率
- 不讲币价
- 不讲 token

3. 先邀请制
- 白名单
- 限额
- 控制补贴

4. 先做内部闭环
- 闲置算力 -> 预算 -> agent 消费
- 不急着做外部流通

## 一句话结论
从运营角度，`stc-mint-agent` 的定位应该是：

> agent 获取可持续预算的底层引擎

如果这个定位站不住，它就会退化成一个既不像 AI 产品、也不像矿工产品的中间态功能。
